# 群聊上下文管理设计方案

> 版本：v2.0
> 适用范围：AgentHub 多 Agent 协作平台 · 群聊模式
> 定位：**分阶段方案**——先定义赛题能落地的精简核心（MVP / P0），把治理、安全、语义冲突等列为可演进项（P1 / P2），既可落地又不丢架构远见。

---

## 0. TL;DR（一页纸看懂最终决定）

1. **保留** v1.1 的四层上下文模型 + 黑板模式 + 横切记忆，这是骨架，方向正确。
2. **采纳** 修订意见里 4 个真正的纠偏（它们修复了 v1.1 的内在矛盾）：
   - 情况 A「完整保留上次会话」→ 改为「**短期热上下文 buffer + 修改前重读产出物**」。
   - 黑板「只存结论」→ 放宽为「**结论 + 提炼后的理由/假设/风险/未决问题**，仍不存原始过程」。
   - 黑板 schema 扩字段（`id/type/status/updated_by/summary` 等），让事实可检索、可失效、可判冲突。
   - 9.2 的「黑板会话流」措辞 → 拆为 **presentation_log（给人）/ blackboard_events（结构化事件）/ blackboard_state（事实快照）** 三分，群聊原文默认不注入 Agent。
3. **新增** 1 个 P0 核心组件：**ContextAssembler（上下文装配器）**——它是「按需拉取」从理念落到代码的唯一入口，并固化**检索优先级**。
4. **冲突处理** MVP 用「**乐观锁（based_on_version）+ 影响范围声明（affected）+ 共享工作区 git 隔离**」；语义冲突检测推迟到 P2。
5. **推迟但预留**：MemoryManager、Contract Watcher、ACL/审批、Prompt Injection 防护、context_trace 可观测性——schema 上预留字段，实现按 P1/P2 推进。
6. **落地**：黑板 + 上下文层是**服务端**能力，复用现有 `AgentSession` + turn + 工作区机制；类型契约进 `packages/shared`。

> 一句话哲学（定稿版）：
> **默认隔离、按需共享；用结构化黑板做唯一真相源而非消息流；Agent 在私有上下文干脏活、只交提炼结论；会话每次重开、靠外部记忆不失忆；改东西以"重读当前产出物"为准，强延续才保留短期热上下文；改动触碰共享契约自动升级为协作。**

## 第一部分：设计哲学

### 1.1 核心矛盾

> Agent 需要"足够的上下文"才能做对事，但上下文越多，成本越高、噪音越大、越容易跑偏。

上下文管理的本质，是在 **信息充分** 与 **信息精简** 之间做动态权衡。

### 1.2 五条根本原则

| 原则 | 含义 |
| --- | --- |
| **默认隔离，按需共享** | 多 Agent 的价值来自上下文隔离；共享是例外，不是默认 |
| **记忆与会话解耦** | 记忆存在外部记忆库，不存在对话历史里；会话临时，记忆持久 |
| **展示与上下文解耦** | "给人看的群聊对话" ≠ "给 Agent 吃的结构化上下文" |
| **产出物即真相源** | 修改时靠"重读当前产出物"，而非"记得当时写了什么" |
| **事实分层、记忆不对抗黑板**（新增） | 检索有明确优先级，私有记忆只能辅助、不能覆盖黑板事实 |

### 1.3 检索优先级

所有上下文装配都遵循同一优先级，冲突时高优先级胜：

```
当前产出物  >  黑板契约 / 决策  >  本次任务上下文  >  Agent 私有记忆  >  历史会话摘要
```

> 这条规则解决了 v1.1 与修订意见反复争论的"记忆 vs 黑板 vs 产出物谁说了算"。

## 第二部分：整体架构

### 2.1 四层上下文模型 + 横切记忆 + 三大组件

```
┌──────────────────────────────────────────────────────────────┐
│  Layer 4 展示层  presentation_log（完整群聊原文）              │
│  - 给人看 / 审计；默认不注入 Agent，仅可被检索工具按需查阅      │
└──────────────────────────────────────────────────────────────┘
                 ▲ 渲染                       ▲ 提炼（Orchestrator/Agent）
                 │                            │
┌──────────────────────────────────────────────────────────────┐
│  Layer 3 全局事实层  Blackboard                                │
│   ├─ blackboard_state  当前事实快照（产出物/决策/契约/任务图）  │
│   └─ blackboard_events 结构化事件流（谁在何时改了哪个事实）     │
└──────────────────────────────────────────────────────────────┘
                 ▲ 读/写（结构化、带版本）
                 │
┌──────────────────────────────────────────────────────────────┐
│  Layer 2 任务上下文层  TaskContext（现场组装，任务结束即销毁）  │
└──────────────────────────────────────────────────────────────┘
                 ▲ 注入
                 │
┌──────────────────────────────────────────────────────────────┐
│  Layer 1 私有工作上下文（草稿/推理/工具原始返回，永不外泄）     │
└──────────────────────────────────────────────────────────────┘

   横切：Agent 长期记忆 agent_memory（每 Agent 私有，带 scope/source/status）

   ── 三大组件（贯穿各层）──
   • MessageRouter   纯机械路由：解析 @、决定去向（不调 LLM）
   • Orchestrator    智能决策：拆解 / 调度 / 协调 / 汇总（调 LLM）
   • ContextAssembler 每次 Agent 调用前，按预算与优先级组装最终上下文
```

### 2.2 各层职责一览

| 层 | 存什么 | 生命周期 | 可见性 | 结构化 |
| --- | --- | --- | --- | --- |
| L1 私有工作上下文 | 推理、草稿、工具返回 | 任务期间，结束即弃 | 仅自己 | 否 |
| L2 任务上下文 | 目标、输入、约束、输出规格 | 单次任务 | 仅被派发 Agent | 是 |
| L3 黑板 state | 产出物、决策、契约、状态、任务图 | 长期，跟随项目 | 所有 Agent 可读 | 是 |
| L3 黑板 events | 事实变更事件 | 长期（可设保留窗口） | Orchestrator/审计 | 是 |
| L4 展示层 | 群聊气泡原文 | 长期 | 用户；Agent 按需检索 | 否 |
| 横切：Agent 记忆 | 经验、约定、踩过的坑 | 长期，跟随 Agent | 仅该 Agent | 是 |

## 第四部分：核心数据模型（MVP 收敛版）

> 原则：MVP 字段尽量少而够用，P1/P2 字段在注释里标出，schema 预留但不强制。

### 4.1 Blackboard（黑板）

```yaml
blackboard:
  project_meta:
    id: "proj_tz"
    name: "时区计算器"
    goal: "..."                       # 项目总目标（Orchestrator 上下文要用）
    tech_stack: ["React", "Vite", "TailwindCSS"]
    status: "development"             # planning | designing | development | done
    workspace_dir: "<群聊共享工作目录>"  # 见第十部分：群聊共享工作区

  artifacts:                          # 产出物索引（结论性事实）
    code:
      id: "art_code_main"
      type: "code"                    # code | document | design | test_report
      path: "src/"
      owner: "FrontendAgent"
      version: 5
      status: "approved"             # draft | proposed | approved | deprecated
      updated_at: "..."
      updated_by: "FrontendAgent"
      summary: "时区计算器前端实现"     # 供"只注入摘要不注入全文"
      # —— P1 字段 ——
      # hash: "...", tags: [...], dependencies: [...]

  decisions:                          # 决策（带状态，可被取代）
    - id: "dec_001"
      content: "时间统一用 UTC 存储，前端转换显示"
      rationale: "避免跨时区歧义"       # ✅ 提炼后的理由，非过程垃圾
      status: "approved"             # proposed | approved | superseded | rejected
      scope: "time_api"
      supersedes: []                  # 取代了哪些旧决策 id
      created_by: "FrontendAgent"
      approved_by: "Orchestrator"
      ts: "..."
      # alternatives_rejected: ["Redux: 过重", ...]   # 可选

  contracts:                          # 共享契约（跨 Agent 协作关键）
    time_api:
      endpoint: "/api/time"
      returns: "UTC ISO string"
      owner: "BackendAgent"           # 谁拥有
      consumers: ["FrontendAgent", "TestAgent"]
      approval_required: true         # 改动需 Orchestrator/用户审批
      version: 3

  task_graph:                         # 任务依赖（DAG）
    - { id: "t1", name: "功能设计", agent: "ProductAgent", deps: [],          status: "done" }
    - { id: "t2", name: "界面设计", agent: "DesignerAgent", deps: ["t1"],      status: "done" }
    - { id: "t3", name: "开发实现", agent: "FrontendAgent", deps: ["t1","t2"], status: "doing" }

  # —— P1 新增对象 ——
  # open_issues: [{ id, content, raised_by, owner, status }]
  # risks:       [{ id, content, severity, owner }]
```

**黑板写入铁律：**
- 只写**结构化字段**，不允许大段自然语言。
- ✅ 可存：产出物索引、决策（含理由/被否方案摘要）、契约、状态、任务图、未决问题、风险、假设、验收标准。
- ❌ 禁存：原始推理、工具调用原始返回、试错草稿、谁对谁说了什么、无结构闲聊。

### 4.2 agent_memory（Agent 长期记忆，私有）

```yaml
agent_memory:
  agent: "FrontendAgent"
  items:
    - id: "mem_001"
      content: "样式统一用 TailwindCSS，不写行内 style"
      type: "convention"             # convention | project_knowledge | lesson | work_done
      scope:                          # ✅ 作用域，避免跨项目污染
        project: "proj_tz"
        module: "frontend"
      source:                         # ✅ 来源可追溯
        type: "blackboard"            # blackboard | self_summary | user
        ref: "project_meta.tech_stack"
      status: "active"               # active | stale | deprecated
      created_at: "..."
      last_used_at: "..."
      # confidence: 0.95              # P1：用于检索排序与信任度
```

> **记忆不对抗黑板**：装配时若 memory 与黑板契约/决策冲突，按检索优先级丢弃 memory 并标 `stale`。

### 4.3 short_term_buffer（短期热上下文，情况 A 专用）

```yaml
short_term_buffer:                    # 短 TTL 热缓存，不是永久会话
  agent: "FrontendAgent"
  task_id: "t3"
  expires_at: "..."                  # 命中窗口（如 5 分钟）后失效
  recent_user_intents: ["开发时区计算器主界面"]
  recent_agent_outputs: ["已完成 src/pages/Calculator.tsx"]
  recent_artifacts: [{ path: "src/pages/Calculator.tsx", version: 5 }]
  mention_index: { "那个切换按钮": "TimezoneSwitchButton" }   # 解析强指代
  unresolved_followups: []
```

> 它只承载"指代解析所需的最小热信息"，**不含**原始推理/工具返回。

### 4.4 TaskContext（任务上下文，现场组装、用完即焚）

```yaml
TaskContext:
  task_id: "t3"
  objective: "给时区计算器加复制结果按钮"
  inputs: [ <从黑板按需拉取的产出物摘要/契约> ]   # 按需，非全量
  constraints: ["不得修改 contracts.time_api，除非升级 Orchestrator"]
  output_spec: "交付 diff + 简短说明"
  mode: "modify_existing"            # new_task | modify_existing
```

## 第五部分：ContextAssembler（P0 核心组件）

> "按需拉取"的唯一落地入口。每次唤起 Agent（含 Orchestrator 派发、用户 @ 修改）都经它组装上下文。

**输入：**
```yaml
ContextAssemblerInput:
  agent_id, task_id, user_request
  target_artifacts: ["src/pages/Calculator.tsx"]
  mode: "modify_existing" | "new_task"
  budget: { max_tokens: 16000 }
```

**输出：**
```yaml
ContextAssemblerOutput:
  system_prompt                      # 角色定义
  task_context                       # L2
  blackboard_facts                   # 相关黑板字段（默认注摘要，不注全文）
  artifact_refs:                     # 让 Agent 自己用文件工具读取
    - { path: "...", version: 5, load_policy: "read_before_edit" }
  memory                             # agent_memory 检索结果（受优先级约束）
  recent_hot_context                 # 仅情况 A 命中时带 short_term_buffer
  tool_access_policy                 # 可用工具/权限（P2 接 ACL）
```

**装配规则（固化检索优先级）：**
1. 先放 system_prompt + TaskContext。
2. 黑板：只拉与 `target_artifacts` / 任务 scope 相关的字段，**默认注 summary**，全文交给 Agent 用工具读。
3. memory：按 scope 过滤，与黑板冲突者丢弃并标 stale。
4. 情况 A 命中 → 附 short_term_buffer；否则不附。
5. **预算超限裁剪顺序**：历史会话摘要 → 低优先级 memory → 非目标产出物摘要 → 保底永远保留：system_prompt + TaskContext + 目标产出物 ref + 相关契约。
6. （P1）产出 `context_trace`：记录注入了哪些 key/memory_id/artifact，及被省略项与原因。

## 第六部分：会话连续性与"再次修改"（核心，已采纳修订）

### 6.1 本质：一个从"强延续"到"无延续"的光谱

用户 `@Agent 改一下 XXX` 时，系统判断这是"接着上次干"还是"独立新活"。

### 6.2 三种情况（情况 A 已按修订意见改写）

| 情况 | 触发特征 | 上下文处理 | 真相源 |
| --- | --- | --- | --- |
| **A 紧接追问（强延续）** | 时间极短(<5min) + 强指代("那个/刚才") + 同产出物 | **short_term_buffer 解析指代 → 重读当前产出物 → 改** | 当前产出物（buffer 仅辅助解析） |
| **B 同产出物新修改（中延续）** | 隔了一段时间、同功能、新小需求 | 重开干净会话 + 黑板产出物摘要 + 相关记忆 + 重读代码 | 当前产出物 |
| **C 完全新任务（无延续）** | 不同产出物、全新需求 | 完全重开 + 仅注通用信息 | 当前产出物 |

> **关键变更（采纳修订意见 #1）**：情况 A 不再"完整保留上次会话（含推理）"。改为热 buffer 解析"那个按钮"指代什么，但**真正动手前仍重读当前产出物**。这样既保留连续性，又不违背"私有上下文永不外泄""产出物即真相源"。

### 6.3 判断与归属

```
用户 @Agent 发起修改
  ├─ 命中短期热窗口(buffer 未过期 + 强指代)? → 情况 A
  ├─ 针对同一产出物/功能模块?              → 情况 B
  └─ 完全不同产出物/新需求?                → 情况 C
```
- **谁判断**：显式 `@Agent` 优先走轻量规则（时间窗口 + 指代检测 + 产出物匹配）；模糊/无 @ 兜底交 Orchestrator（LLM）。
- **默认重开 + 注入记忆，延续是需要被识别出来的例外。**

### 6.4 统一收尾（任何情况改完都做）

```
1. 走产出物流程：read → plan → patch → test → summarize
2. 更新黑板：artifacts.<x>.version += 1, updated_by/at, 必要时写 decisions(带 supersedes)
3. 提交 memory_candidate（经轻量去重写入 agent_memory）
4. report_completion 给 Orchestrator（若属协作任务）
5. 会话：A 可保持热窗口；B/C 一段时间无追问 → 提炼记忆 → 关闭
```

### 6.5 修改触碰共享契约 → 自动升级为协作

```
@FrontendAgent 接口改成返回本地时间，不要 UTC
  → 命中 contracts.time_api（approval_required=true, 非该 Agent owner）
  → 禁止单方改写，上报 Orchestrator
  → Orchestrator 派生 BackendAgent 配合任务 + 通知 consumers(TestAgent)
```
- MVP：写入共享契约字段时**被动检测**并上报。
- P1：Preflight 主动预检（改之前先判断 may_affect）+ Contract Watcher（owner/consumers 自动通知）。

## 第七部分：任务调度与冲突处理

### 7.1 任务图驱动（DAG）

- 无依赖 / 依赖已满足 → 可立即派发；多个就绪 → **并行派发**。
- 任务完成 → 更新 `task_graph.status` → 检查解锁下游。

### 7.2 失败降级（赛题要求）

| 失败 | 降级策略 |
| --- | --- |
| 子 Agent 单轮失败/超时 | Orchestrator 捕获 `done(success=false)`，重试 1 次；仍失败 → 标记任务 `failed`，在聊天流汇报并请用户决策 |
| 依赖任务失败 | 下游任务挂起为 `blocked`，不连锁假装成功 |
| 找不到合适 Agent | 退化为"由最通用 Agent 处理"或回问用户 |

### 7.3 代码冲突处理（赛题要求）

**MVP 三件套（廉价、够用）：**
1. **乐观锁**：Agent 写黑板/产出物时声明 `based_on_version`；当前版本不符 → 拒绝，要求重读后重试。
2. **影响范围声明**：Agent 提交结果时声明 `affected: { artifacts, contracts, decisions }`，便于检测交叠。
3. **共享工作区 + git 隔离**：群聊有一个共享工作目录（见第十部分）；**并行任务各自在 git worktree/分支干活**，完成时合并；合并冲突 → 上报 Orchestrator → 由专责 Agent 或用户裁决。

**P2（推迟）：语义冲突检测**——不同 key 但语义打架（如 contracts 说 UTC、某 decision 说后端直接返回本地时间）。需要规则/LLM 校验是否违反现有 contract、是否与 approved decision 冲突、是否让下游结果失效。成本高、边际收益有限，赛题阶段不做。

## 第八部分：角色编排（MessageRouter / Orchestrator）

### 8.1 职责边界

| 组件 | 职责 | 调 LLM |
| --- | --- | --- |
| **MessageRouter** | 纯机械：解析 @、决定消息去向、原文写入 presentation_log | ❌ |
| **Orchestrator** | 智能：复杂度判断、拆解、调度、协调、汇总 | ✅ |

> 复杂度判断属"智能决策"，归 Orchestrator；MessageRouter 不做任何"理解"。

### 8.2 路由规则（修正 9.2 措辞）

| 用户输入 | 路由行为 |
| --- | --- |
| `@SingleAgent <msg>` | 发给该 Agent；原文进 **presentation_log**（不进黑板事实流） |
| `@AgentA @AgentB <msg>` | 发给 A、B；通知 Orchestrator 轻量协调；轻聊走成员真实轻量 turn，交付走任务 |
| `@Orchestrator <msg>` | 强制 Orchestrator 介入；任务消息生成计划，非任务消息由 Orchestrator 直接回复；需要成员本人轻量回应时走 `memberTurns` |
| `<msg>`（无 @） | 默认交 Orchestrator 判断复杂度；任务消息分发，非任务消息由 Orchestrator 直接回复；需要成员本人轻量回应时走 `memberTurns` |

> **关键变更（采纳修订意见 #10）**：删除原"记入黑板会话流"。用户原文统一进 `presentation_log`（给人/审计）；只有被提炼的有价值部分，才由 Orchestrator/Agent 写成 `blackboard_event` 或 `task_context`。Agent 默认不吃群聊原文，需要时用检索工具按需查阅。

### 8.3 Orchestrator 自身上下文（采纳修订意见 #9，防"上下文黑洞"）

Orchestrator 拿的是**项目控制面板**，不是全量历史：
```yaml
orchestrator_context:
  project_goal                       # blackboard.project_meta.goal
  active_task_graph                  # 仅 doing/ready/blocked 任务
  open_issues / current_blockers
  recent_user_intents                # 最近 N 条用户意图（非全文聊天）
  blackboard_summary                 # 状态/契约/决策摘要
  agent_status                       # 各 Agent busy/idle/last_output
```
> 它不吃完整群聊记录，也不吃任何 Agent 的私有过程。

### 8.4 协作主流程

```
用户消息 → MessageRouter(解析@, 原文进 presentation_log)
  → Orchestrator: 复杂度判断
       简单 → 选最合适 Agent，单任务派发
       复杂 → 拆解写入 blackboard.task_graph(用户可在任务面板编辑)
  → 按 DAG 调度，dispatch_agent(经 ContextAssembler 组装上下文)
  → 子 Agent 在共享工作区/隔离分支干活, 完成 report_completion + 更新黑板
  → Orchestrator 聚合产出, 在聊天流汇报(群聊成员依次"发言")
```

## 第九部分：安全（MVP 最小原则 + 演进）

| 项 | MVP（P0 原则） | 演进 |
| --- | --- | --- |
| Prompt Injection | 产出物/网页/文件内容一律视为**不可信数据**，不得覆盖 system/Orchestrator 指令；装配时数据与指令分区 | P2：可信边界标注、内容消毒、注入检测 |
| 权限控制 | 约定式：契约 owner 才能改；非 owner 改触发升级 | P2：黑板 + 工具 ACL（按 Agent 角色） |
| 审批机制 | 高危动作（删文件/改契约/改 schema/部署/产生费用）原则上需 Orchestrator 或用户确认 | P2：正式审批流 + 审计 |

## 第十部分：落地到 AgentHub 架构

> 复用现有单聊机制（见 `agent-single-chat-spec.md`），群聊是其自然扩展，**不重造轮子**。

### 10.1 概念映射

| 上下文概念 | 落到 AgentHub |
| --- | --- |
| 群聊 | 新增 `GroupChat`：容纳多个成员 Agent + 一块黑板 + 一份 presentation_log |
| 群成员的一次干活 | 复用现有 `AgentSession` + turn（游离后台任务 + Redis Stream 围观），每成员一个 session |
| 私有工作上下文(L1) | 即 turn 内 SDK 句柄的运行步骤（thinking/tool，已落 `agent_message_step`），天然私有 |
| 产出物访问层 | **复用子 Agent 自带文件工具**（Claude/Codex）；read→plan→patch→test 由 agent runtime 完成 |
| 共享工作区 | 群聊一个共享 `workspace_dir`；并行任务用 **git worktree/分支** 隔离，完成合并（对应单聊的 per-session `workingDirectory`，群聊升级为共享 + 隔离分支） |
| 黑板 / 记忆 | **服务端新存储**：`blackboard_state`/`decisions`/`contracts`/`task_graph`/`agent_memory` 落 MySQL；`blackboard_events` 可走 Redis Stream（与 turn 事件流同款机制） |
| presentation_log | 群聊 UI 消息历史（类比 `agent_message`，但多发言者 + 系统/Orchestrator 卡片） |
| ContextAssembler / Orchestrator / MessageRouter | 服务端新模块，置于 `apps/server/src/multiagents/`（与单聊同域） |

### 10.2 需要新增的东西（概要，待 plan 阶段细化）

- **`packages/shared`**：群聊相关协议类型——`GroupChat`、`Blackboard`（state/artifact/decision/contract/task_graph）、`AgentMemoryItem`、`TaskContext`、`ContextAssemblerInput/Output`、Orchestrator 派发/汇报 envelope。所有跨端契约在此定义。
- **`apps/server/src/multiagents/`**：`GroupChatModule`、`BlackboardService`、`ContextAssembler`、`Orchestrator`、`MessageRouter`、`dispatch_agent`/`report_completion` 协作端点 + 群聊事件 SSE。
  - ⚠️ 动 `apps/server` 前必读 `apps/server/CLAUDE.md`；凡涉及前后端契约，前后端两份 CLAUDE.md 都要读。
- **`apps/desktop`**：把 ChatList 里"创建群聊"占位变实；群聊视图渲染多 Agent 气泡 + 任务面板（可编辑 task_graph）+ Orchestrator 汇报卡 + 黑板/产出物面板（复用设计系统 ui/ 组件与 Tailwind tokens，不引入裸 hex / 手搓按钮）。

### 10.3 复用要点

- turn 的"后台游离 + 多端围观"机制对群聊每个成员一样适用：多成员并行时就是多条并行 turn，各自一条 Redis Stream。
- 并发互斥锁、abort、超时兜底、boot 回收等单聊已有能力直接继承到每个成员 session。
